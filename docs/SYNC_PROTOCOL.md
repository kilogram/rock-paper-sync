# reMarkable Sync v3 Protocol - Reverse Engineered

## Overview

This document captures critical findings from reverse engineering the reMarkable Sync v3 protocol through device analysis and live testing with rm_cloud.

## Critical Discovery: hashOfHashesV3 Algorithm

### The Problem

Documents uploaded to rm_cloud were appearing in the cloud but not syncing to the device. Error logs showed:

```
Downloaded invalid schema hash for {doc-uuid}:
  serverHash=ec45daa6...
  calculated=0c07c238...
  schema=SupportedSchemaVersion::hashOfHashesV3
```

### The Investigation

We reverse engineered the algorithm by:
1. Creating a test document on the device
2. Downloading its index structure from rm_cloud
3. Testing multiple hash calculation methods
4. Matching the device's calculated hash

### The Algorithm

**hashOfHashesV3 = SHA256 of concatenated binary file hashes (sorted by filename)**

```python
# Sort file entries by entry_name
sorted_entries = sorted(file_entries, key=lambda e: e.entry_name)

# Convert hex hashes to binary and concatenate
file_hashes_binary = b''.join(
    bytes.fromhex(entry.hash) for entry in sorted_entries
)

# Calculate SHA256
hash_of_hashes = hashlib.sha256(file_hashes_binary).hexdigest()
```

### Implementation Requirements

1. **Root Index**: Use `hashOfHashesV3` as the document's hash in the root index
2. **Document Index**: Upload the document index blob under BOTH:
   - Its content hash (SHA256 of the index file itself)
   - The `hashOfHashesV3` (so the device can download it)

```python
# Upload index under content hash (for internal use)
doc_index_hash = sha256(index_content)
upload_blob(doc_index_hash, index_content)

# Upload AGAIN under hashOfHashesV3 (for device download)
hash_of_hashes = calculate_hash_of_hashes(file_entries)
upload_blob(hash_of_hashes, index_content)

# Use hashOfHashesV3 in root index
root_entry = BlobEntry(hash=hash_of_hashes, ...)
```

## CRDT Format Requirements

### Static Timestamps

The device uses **static CRDT counters**, not incrementing timestamps:

```json
{
  "id": "page-uuid",
  "idx": {
    "timestamp": "1:2",  // STATIC - represents logical clock position
    "value": "ba"
  },
  "template": {
    "timestamp": "1:1",  // STATIC
    "value": "Blank"
  }
}
```

**Wrong**: Using Unix timestamps like `"1:1763239068"`
**Right**: Using small static counters like `"1:1"`, `"1:2"`, `"1:3"`

### The "modifed" Field (Intentional Typo!)

Content changes are signaled via the `modifed` field (note the typo):

```json
{
  "id": "page-uuid",
  "idx": {...},
  "modifed": "1763239413854",  // Unix timestamp in milliseconds
  "template": {...}
}
```

**Critical**: The field name is `"modifed"` (missing 'i'), NOT `"modified"`

This field contains the current Unix timestamp in milliseconds and is the primary signal to xochitl that content has changed.

## Required Files

Each document needs these files uploaded to sync properly:

### 1. `.metadata` - Document metadata
```json
{
  "visibleName": "Document Title",
  "type": "DocumentType",
  "parent": "parent-uuid-or-empty",
  "lastModified": "1763239068000",
  "version": 1,
  "synced": true,
  "modified": false,
  "deleted": false
}
```

### 2. `.content` - CRDT page structure
```json
{
  "formatVersion": 2,
  "fileType": "notebook",
  "pageCount": 1,
  "cPages": {
    "pages": [
      {
        "id": "page-uuid",
        "idx": {"timestamp": "1:2", "value": "ba"},
        "modifed": "1763239413854",
        "template": {"timestamp": "1:1", "value": "Blank"}
      }
    ],
    "lastOpened": {"timestamp": "0:0", "value": ""},
    "original": {"timestamp": "0:0", "value": -1},
    "uuids": [{"first": "device-uuid", "second": 1}]
  }
}
```

### 3. `.local` - Empty JSON (required by xochitl)
```json
{}
```

**Critical**: This file must exist for xochitl to recognize the document, even though it's empty.

### 4. `{page-uuid}.rm` - Binary page content
Binary v6 format generated by rmscene library.

## Page UUID Reuse

When updating documents, **reuse existing page UUIDs** to avoid CRDT merge conflicts:

```python
# Fetch existing page UUIDs from cloud
existing_uuids = get_document_page_uuids(doc_uuid)

# Reuse them when generating updated document
for i, page in enumerate(pages):
    if i < len(existing_uuids):
        page_uuid = existing_uuids[i]  # Reuse
    else:
        page_uuid = generate_uuid()    # New page
```

## Document Deletion

Remove documents by:
1. Download current root index
2. Filter out the document entry
3. Create new root index without the document
4. Upload new root index blob
5. Update root pointer with broadcast=True

```python
# Get current root
root_entries = get_root_documents()

# Remove document
new_entries = [e for e in root_entries if e.entry_name != doc_uuid]

# Upload new root
new_root_hash = upload_index(new_entries)
update_root(new_root_hash, current_gen, broadcast=True)
```

## Key Takeaways

1. **hashOfHashesV3** is NOT the index hash - it's a merkle-tree-like hash of file hashes
2. Document index must be stored under **both** hashes
3. CRDT timestamps are **static** - updates signaled via `modifed` field
4. The `modifed` typo is **intentional** and must be preserved
5. `.local` file is **required** even though empty
6. Page UUIDs should be **reused** on updates
